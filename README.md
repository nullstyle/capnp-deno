# capnp-deno (scaffold)

`capnp-deno` is a Deno-side runtime scaffold for driving the Cap'n Proto RPC
core over WASM.

Current scope:

- wraps low-level WASM ABI exports from `docs/deno_wasm_abi_draft.md`,
- provides `WasmPeer` for frame push/pop,
- provides `RpcSession` to wire a Deno transport to a `WasmPeer`,
- provides `WasmSerde` to call generated schema-specific JSON serde exports,
- includes real transports:
  - `MessagePortTransport` (worker/message-channel),
  - `WebSocketTransport` (message-oriented),
  - `TcpTransport` + Cap'n Proto stream framing (`CapnpFrameFramer`).
- includes real-wasm tests for:
  - generated-schema `Person` serde (`capnp_example_person_to_json` /
    `_from_json`),
  - bootstrap + call success/error fixtures against the real peer state machine
    (success path uses test-only raw wasm hook `capnp_peer_set_bootstrap_stub`).
- includes an early `capnpc-deno` scaffold (`tools/capnpc-deno/`) that parses
  `CodeGeneratorRequest` and emits TypeScript type/codec stubs.

## Layout

- `src/abi.ts`: typed ABI surface + memory/error helpers.
- `src/wasm_peer.ts`: peer handle lifecycle and frame pump helpers.
- `src/session.ts`: transport integration with ordered inbound pumping.
- `src/serde.ts`: wrapper for calling wasm serde exports (`to_json` /
  `from_json`).
- `src/load.ts`: wasm instantiation helper that returns a ready `WasmPeer`.
- `src/rpc_wire.ts`: minimal RPC wire builders/parsers (bootstrap/call/return).
- `src/rpc_client.ts`: `SessionRpcClientTransport` + in-memory harness transport
  for fixture-style client driving of `RpcSession`.
- `src/framer.ts`: stream framing parser for Cap'n Proto RPC over TCP.
- `src/transports/`: MessagePort, WebSocket, and TCP transport adapters.
- `tests/`: fake-WASM based tests that validate host logic without compiled Zig
  exports.
- `tests/fixtures/rpc_frames.ts`: deterministic wire fixtures generated by
  `zig build gen-rpc-fixtures`.

## Core Contract

After each inbound frame is pushed, host code must drain all outbound frames
until empty.

`RpcSession` enforces this by:

1. calling `peer.pushFrame(inbound)`,
2. sending every returned outbound frame through the transport in order.

## Server Runtime Helper

For wasm host-call bridge deployments, use `RpcServerRuntime` to combine:

1. `RpcSession` inbound/outbound transport pumping,
2. `RpcServerBridge` dispatch handling, and
3. automatic `capnp_peer_pop_host_call` -> response pumping after each inbound
   frame.

`RpcServerRuntime` includes optional guardrails against runaway host-call pump
loops:

- `maxCallsPerInboundFrame`: bounds host-call work per inbound frame,
- `maxCallsTotal`: lifetime budget before protection trips,
- `failOnLimit`: fail-fast (`true`) or warn-and-stop pump (`false`).

Host-call frame ownership is released through `capnp_peer_free_host_call_frame`
when that export is available.

Use `onWarning` with `failOnLimit: false` to surface operational alerts.

## Example

```ts
import { RpcSession, type RpcTransport, WasmPeer } from "./mod.ts";

const transport: RpcTransport = {
  async start(onFrame) {
    // Wire network callback to onFrame(...)
  },
  async send(frame) {
    // Write bytes to network
  },
  async close() {},
};

// instance: WebAssembly.Instance from your compiled capnp wasm module
const peer = WasmPeer.fromInstance(instance, { expectedVersion: 1 });
const session = new RpcSession(peer, transport);
await session.start();
```

JSON serde wrapper usage (requires generated exports in wasm module):

```ts
import { WasmSerde } from "./mod.ts";

const serde = WasmSerde.fromInstance(instance, { expectedVersion: 1 });
const codec = serde.createJsonCodec<{ id: number }>({
  toJsonExport: "capnp_example_message_to_json",
  fromJsonExport: "capnp_example_message_from_json",
});

const bytes = codec.encode({ id: 42 });
const value = codec.decode(bytes);
```

Auto-discover generated serde codec pairs:

```ts
import { WasmSerde } from "./mod.ts";

const serde = WasmSerde.fromInstance(instance, { expectedVersion: 1 });
const codecs = serde.listJsonCodecs();
// [{ key: "example_person", toJsonExport: "..._to_json", fromJsonExport: "..._from_json" }]

const personCodec = serde.createJsonCodecFor<{
  name: string;
  age: number;
}>({ key: "example_person" });
```

## Observability (Deno OpenTelemetry)

`capnp-deno` now exposes an observability hook on `RpcSession` and transport
options. For Deno OpenTelemetry, use `createDenoOtelObservability(...)`:

```ts
import {
  createDenoOtelObservability,
  MessagePortTransport,
  RpcSession,
} from "./mod.ts";

const observability = createDenoOtelObservability({
  instrumentationName: "@capnp/deno",
});

const transport = new MessagePortTransport(port, { observability });
const session = new RpcSession(peer, transport, { observability });
```

To export telemetry from Deno, configure OTLP env vars (for example):

```sh
OTEL_DENO=true \
OTEL_SERVICE_NAME=capnp-deno-dev \
OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf \
OTEL_EXPORTER_OTLP_ENDPOINT=http://127.0.0.1:4318 \
deno test --allow-net --allow-read
```

See Deno OpenTelemetry docs for current limitations and stability notes.

## Reconnect Helpers (Opt-In)

Reconnect behavior is explicit and policy-driven. Nothing reconnects unless you
use reconnect helpers.

```ts
import {
  connectTcpTransportWithReconnect,
  createExponentialBackoffReconnectPolicy,
} from "./mod.ts";

const transport = await connectTcpTransportWithReconnect("127.0.0.1", 7000, {
  transport: { connectTimeoutMs: 500 },
  reconnect: {
    policy: createExponentialBackoffReconnectPolicy({
      maxAttempts: 5,
      initialDelayMs: 100,
      maxDelayMs: 2_000,
      factor: 2,
      jitterRatio: 0.2,
    }),
  },
});
```

For session startup flows, use `createRpcSessionWithReconnect(...)` to retry
transport connection before creating/starting `RpcSession`.

For generated RPC clients, use `ReconnectingRpcClientTransport` to add
established-session reconnect behavior for call paths:

- retries connect using your reconnect policy,
- retries in-flight call once by default on reconnectable errors,
- if the call used the current bootstrap capability, it re-runs bootstrap and
  remaps capability before retry,
- for non-bootstrap capabilities, provide `remapCapabilityOnReconnect(...)` when
  capability IDs are expected to change across reconnects,
- `finish`/`release` failures are surfaced as non-retriable semantic errors
  because IDs are connection-scoped.

## Development

```sh
deno task fmt
deno task lint
deno task check
deno task check:tools
deno task test:codegen
deno task test:unit
deno task test:integration
deno task test:real
```

Codegen usage:

```sh
deno task codegen generate --src tests/fixtures/schemas --out generated
deno task codegen generate --schema tests/fixtures/schemas/person_codegen.capnp --out generated
deno task codegen generate --request-bin path/to/request.bin --out generated
```

Useful options:

- `--layout schema|flat` controls output paths (`schema` preserves schema
  directories, `flat` keeps legacy single-directory output).
- `--no-barrel` skips generated `mod.ts` barrel exports.
- `--plugin-response` writes `CodeGeneratorResponse` bytes to stdout instead of
  writing files to disk.
- `--config path/to/capnpc-deno.toml` loads generation defaults from TOML.
- `--no-config` ignores auto-discovered `capnpc-deno.toml`.
- positional schemas are supported in `generate` mode:
  `deno task codegen generate schema/foo.capnp`.
- plugin-style stdin mode is supported with positional out dir:
  `capnp compile -I schema -o- schema/foo.capnp | deno task codegen generated`.
- direct plugin mode is supported via wrapper script:
  `capnp compile -I schema -o ./scripts/capnpc-deno:generated schema/foo.capnp`.

Example `capnpc-deno.toml`:

```toml
src = ["schema"]
out_dir = "generated"
import_paths = ["schema", "vendor/capnp"]
layout = "schema"
emit_barrel = true
plugin_response = false
```

Current scaffold limitations:

- generated codecs currently target supported field shapes (primitives, enums,
  text/data, lists, nested structs, unions/groups, interface pointers, and
  anyPointer null/interface variants),
- `capnpc-deno` now emits `_rpc.ts` and `_meta.ts` modules, but RPC generation
  is still scaffold-level (client + generated server dispatch emit implemented;
  full capability lifecycle/bridge integration is pending),
- `SessionRpcClientTransport` supports non-empty payloads and cap-table
  metadata, but deeper promise pipelining and fully automatic reconnect loops
  are still in progress,
- plugin invocation defaults to file-emitting mode (cwd output directory);
  `CodeGeneratorResponse` stdout mode is opt-in via `--plugin-response`.

Regenerate RPC fixture constants:

```sh
zig build gen-rpc-fixtures > tests/fixtures/rpc_frames.ts
```

## Real WASM Smoke

Build and copy the wasm artifact into `generated/` and run a local smoke check:

```sh
deno task verify:real
```

By default `scripts/build_wasm.sh` auto-detects the monorepo root. If
`capnp-deno` is in its own repository, set:

```sh
CAPNPC_ZIG_ROOT=/path/to/capnpc-zig deno task build:wasm
```

Real-wasm examples/tests load `generated/capnp_deno.wasm` unless an explicit
path argument is provided.
